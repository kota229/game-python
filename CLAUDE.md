## Git / GitHub 運用ルール（重要）
- **コミット**: 意味のある単位でこまめにコミットする。最低でも各マイルストーン
  （PLAN.md の M1, M2, ...）完了時には必ずコミットすること
- **コミットメッセージ**: 日本語でよい。1行目に要約、必要なら本文に詳細
  （例: `M2完了: ブロック実行エンジンとクリア判定を実装`）
- **push**: 各マイルストーン完了時に必ず GitHub（origin）に push すること。
  これが最重要のバックアップとなる
- **ブランチ戦略**:
  - `main` は常に「動く状態」を保つ（テストが通らないコードを main に入れない）
  - フェーズ1（M1〜M3）は `main` で直接作業してよい
  - フェーズ2の各マイルストーンは `feature/m4-python-editor` のような
    ブランチを切って作業し、完了時に Pull Request を作成する
    （`gh pr create` を使用。マージはユーザーが GitHub 上で行うので待つこと）
- **Issue対応**: ユーザーから「Issue #N を対応して」と指示されたら、
  `gh issue view N` で内容を確認し、ブランチを切って対応し、
  PR の説明文に `Closes #N` を含めること
- **してはいけないこと**:
  - `git push --force` の使用（履歴が壊れるため禁止）
  - `main` への直接 push（フェーズ2以降）
  - トークンやAPIキーをコードやコミットに含めること
- **CI**: GitHub Actions のテストが失敗した場合は、原因を修正してから次の作業に進む

## 技術方針
- HTML5 + TypeScript(または JavaScript)のシングルページアプリ
- M1 の調査結果に基づき、実績あるOSS部品（例: ブロックUIに Blockly、
  エディタに CodeMirror 等）の採用を積極的に検討する。車輪の再発明より
  実績部品の活用を優先し、採用判断は DECISIONS.md に記録すること
- Androidタブレットのブラウザ / WebView で動作すること（PWA対応推奨）
- **GitHub Pages でそのまま公開できる構成にすること**（これが主要な納品経路。
  ビルド不要の静的構成にするか、Actions でビルド→Pages デプロイを自動化する）
- Capacitor で APK 化できるプロジェクト構成も維持する
  （環境的にAPKビルド不可の場合は、README にビルド手順を明記すれば可）
- Python 実行部分は Pyodide（ブラウザ内Python）を第一候補とする。
  導入が困難な場合は自作の簡易インタプリタで代替し、DECISIONS.md に記録する
- オフラインで完結すること（初回ロード後は外部通信を必要としない）
- 進捗データは localStorage に保存する

## 既存OSSの調査・活用ルール（重要）
- 開発に先立ち、GitHub上の類似OSS学習ゲームを徹底調査し、設計・UI・演出・
  カリキュラムの知見を docs/research.md に蓄積すること（PLAN.md の M1）
- 参照用にクローンした他者のリポジトリは `reference/` に置き、
  `.gitignore` で自分のリポジトリから除外すること
- **ライセンス遵守のルール**:
  - 他者のコード・ライブラリを利用する前に、必ずそのリポジトリの LICENSE を
    実際に確認し、docs/research.md に記録すること
  - 利用可: MIT / Apache-2.0 / BSD / MPL-2.0 等の寛容ライセンス
    （ライブラリとしての利用、および出典明記のうえでのコード参考・一部利用）
  - 原則回避: GPL / AGPL 系のコードの取り込み（プロジェクト全体のライセンスに
    影響するため。どうしても必要な場合は取り込まず、設計思想の参考にとどめる）
  - 注意: コードがOSSでも、画像・音・キャラクター等のアセットは別ライセンスの
    ことが多い（例: CodeCombat はコードMITだがアートは非公開ライセンス）。
    **アセットの流用は一切禁止**とし、音・画像はすべて自作する
    （Web Audio API によるプログラム生成、SVG/Canvas による自作グラフィック）
  - 利用したOSSはすべて `CREDITS.md` に「名称 / リポジトリURL / ライセンス /
    利用箇所」を記載し、ライセンスが要求する表記（著作権表示等）を満たすこと
- Lightbot / Human Resource Machine 等の商用ゲームは、公開情報からの
  設計思想・レベルデザインの参考にとどめること（コード・素材の流用不可）
