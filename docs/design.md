## 1. コンセプトと題材
ロボット「コード」をマス目マップ上でゴールへ導く／ジェムを集める課題を通じて、
**順次実行 → ループ → 条件 → 変数・関数・リスト** を、ビジュアルブロックから
本物のPythonまで **4段階で橋渡し** しながら学ぶ。1ステージ1〜3分のテンポ。

設計の下敷き（`docs/research.md` より、いずれもコード流用せず設計参考）:
- **Hedy**: 構文を一度に見せず段階導入する「gradual language」思想 → 4段階カリキュラムの粒度。
- **Blockly Games（Maze）**: レベルごとに使えるブロックを解放し、ブロック数制限で最適解を促す難易度カーブ。
- **Reeborg's World**: マス目ロボット課題を **データで宣言的に定義** する設計 → `data/stages.js` スキーマ。
- **CodeCombat**: スター評価・アンロック等の報酬設計。

## 3. 画面遷移

```
┌──────────┐   スタート    ┌──────────────┐  ステージ選択  ┌────────────┐
│ タイトル  │ ───────────▶ │ ステージマップ │ ───────────▶ │ プレイ画面  │
│  画面     │ ◀─────────── │ (進捗・スター) │ ◀─────────── │            │
└──────────┘    もどる      └──────────────┘   クリア/戻る  └─────┬──────┘
                                    ▲                              │ 実行
                                    │      クリア演出（★評価）      ▼
                                    └──────────────────────┐ ┌──────────────┐
                                                            └─│ クリアダイアログ│
                                              つぎへ / もう一度  └──────────────┘
```

- **タイトル画面**: タイトル、「はじめる」「つづきから」、音ON/OFF。
- **ステージマップ**: ステージをカードで並べ、クリア済みは獲得スターを表示。未解放は施錠（前段クリアで解放）。
- **プレイ画面**: 左にマップ（Canvas）、右にブロックパレット＋プログラム組み立て欄、上にステージ説明、
  下に「じっこう」「リセット」「ヒント」。実行するとロボットが1命令ずつアニメーションで動く。
- **クリア演出**: 効果音＋スター1〜3個のアニメ表示。「つぎへ」「もういちど」。

## 4. データ構造

### 4.1 ステージ定義（`data/stages.js` の1要素）
```js
{
  id: 'p1',                       // 一意なID
  title: 'ゴールまで まっすぐ',    // 子ども向けタイトル（ふりがな相当のやさしい表記）
  concept: 'じゅんじつ実行',       // 学ぶ概念（説明用）
  grid: { cols: 5, rows: 3 },     // 盤面サイズ
  robot: { x: 0, y: 1, dir: 1 },  // 初期位置と向き（0上/1右/2下/3左）
  goal: { x: 4, y: 1 },           // ゴール（無い場合は null 可）
  gems: [{ x: 2, y: 1 }],         // 集めるジェム（任意）
  walls: [{ x: 3, y: 0 }],        // 通れないマス（任意）
  goalType: 'reach'|'collect'|'collect_and_reach',
  allowed: ['forward','left','right','collect','repeat','if','while'], // パレットに出すブロック
  stars: { three: 4, two: 6 },    // ブロック数による★3/★2閾値（任意）
  tutorial: '説明文',
  hints: ['ヒント1','ヒント2','答えに近いヒント'],  // 3段階
  solution: [ /* ブロックノード配列（模範解答） */ ],
}
```

### 4.2 ブロックノード（プログラムの内部表現）
```js
{ type:'forward' }                                  // 前進
{ type:'left' } | { type:'right' }                  // 左右回転
{ type:'collect' }                                  // 拾う
{ type:'repeat', times:n, body:[...] }              // n回くりかえし
{ type:'if', cond, body:[...], else:[...]? }        // 条件分岐
{ type:'while', cond, body:[...] }                  // 条件ループ
// cond: 'path_ahead' | 'blocked' | 'on_gem' | 'at_goal'
```

### 4.3 進捗データ（localStorage: `codequest.progress.v1`）
```js
{ "p1": { cleared:true, stars:3 }, "p2": { cleared:true, stars:2 }, ... }
```
過去より良いスターのみ更新。ステージ解放判定は「直前ステージが cleared か」で行う。

## 5. コード実行エンジンの方式

- **トレースベース実行**（DOM非依存の純粋モジュール）: `interpreter.run(world, program)` が
  プログラムを解釈し、**1アクション＝1イベント** の配列（trace）と最終ワールドを返す。
  UI はこの trace を一定間隔で1つずつ再生してロボットを動かす（Scratch VM のステップ実行モデルを参考）。
- **無限ループ対策**: `MAX_STEPS`（500）で while の暴走を打ち切りエラー扱い。
- **クリア判定**: `rules.checkGoal(world, stage)` と `rules.evaluate(...)`（★評価）。純粋関数として
  Node で自動テスト可能（`test/engine.test.js`, `test/stages.test.js`）。
- **Python実行（段階4, M5で実装）**: Pyodide を第一候補、Skulpt を軽量代替として実機ロード重量を
  スパイクで比較して確定（`DECISIONS.md`）。Stage 4 のロボット操作は `robot.forward()` 等の
  ブリッジAPIをサンドボックスに注入し、ユーザーPythonの実行結果を同じ trace 形式に落として再生する。

## 8. M6 実装状況（31ステージ）と難易度カーブ

M6 で合計 **31ステージ** を実装（block 10 / bridge 5 / fill 8 / free 8）。全ステージの解答は
`test/stages.test.js` で「実際にクリアできる」ことを自動検証している（★3含む）。

### 難易度カーブ（新出概念は原則1ステージ1つ）
| 区間 | ステージ | 積み上がる概念 |
|---|---|---|
| 段階1 ブロック | p1→p5 | 順次 → 回転 → 回収 → repeat → repeat+回転 |
| 〃 | p6→p10 | 迷路（手順分解）→ if → if/else → while → while+if+回収（総合） |
| 段階2 bridge | b1→b5 | ブロックの横に対応Pythonを表示。repeat=for / if / while の対応を「見て」学ぶ |
| 段階3 fill | f1→f8 | range数 → メソッド名 → while条件 → if条件 → 複数穴。だんだん埋める箇所が増える |
| 段階4 free | c1→c8 | for → while → for+if回収 → while+if → ジグザグ → L字 → if/else → 総合 |

各段の境目に橋渡し（同じ課題を次の形式でも解く）を置き、急なジャンプを避けている。
条件・ループは「ブロック(段1)→見る(段2)→埋める(段3)→書く(段4)」の順で4回ふれる設計。

### 段階4+: 計算・データ（mode:'compute'）— M6.5 で追加

**これで合計39ステージ**（block10/bridge5/fill8/free8/compute8）。REQUIREMENTS が挙げる Python の
範囲（変数・四則演算・if/elif/else・for/while・def・リスト・文字列操作）を**すべて網羅**した。
`pyRun` の対応範囲は DECISIONS.md（M6.5）参照。Pyodide は不要（サブセットで要件充足・軽量・
オフライン・Node での自動検証を維持）。
